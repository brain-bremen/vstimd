# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**vstimd** is a Rust rewrite of the C++ StimServer visual stimulus server, combined with ideas from the VStim project. The Rust server binary is named `vstimd`; the overall project (server + Python client + tools) is vstimd.

**Goal:** A remote PsychoPy-compatible visual stimulus server with enhanced features for neuroscience research.

The original StimServer used MFC/C++/Direct3D11 with a client/server architecture over Windows named pipes (binary protocol). This project ports that architecture to Rust, targeting Linux as the primary deployment platform, replacing named pipes with ZeroMQ for cross-platform IPC, and using Vulkan for GPU rendering.

The `extern/` directory contains git submodules for external references: `extern/StimServer/` holds the original C++ reference implementation and `extern/psychopy/` holds PsychoPy for stimulus design reference.

## Build & Run

```bash
# Build (both backends compiled on Linux, desktop-only on Windows)
cargo build --release
cargo test
cargo clippy

# Run — automatic mode detection
cargo run --release
# On Linux without display server: uses DRM/Vulkan (bare-metal)
# On Linux with X11/Wayland: uses winit/Vulkan (desktop)
# On Windows: always uses winit/Vulkan (desktop)

# Window mode options (desktop mode only, default is fullscreen)
cargo run --release -- --windowed 1280x720

# Null renderer — ZMQ server only, no display (also: VSTIMD_NULL=1)
cargo run --release -- --null

# Python client
cd client/python
uv sync
uv run examples/flash_rects.py
```

## Stack

- **ash 0.38** — Vulkan bindings (both backends)
- **ash-window 0.13** — Vulkan surface creation from winit window handle
- **winit 0.30** — Window + event loop (`ApplicationHandler` trait, desktop mode)
- **egui 0.34.2 + egui-winit 0.34.2** — Debug overlay (desktop mode; custom Vulkan renderer in `render/vk/egui/`)
- **input 0.9** — libinput keyboard reader (DRM/console mode, Linux only)
- **lyon_tessellation 1.0** — CPU tessellation of 2-D paths into triangle meshes
- **prost 0.13** — Protobuf encode/decode; stubs generated by `build.rs` via `prost-build`
- **naga 23** — WGSL → SPIR-V compilation at build time (grating shader)
- **zeromq 0.4** — Pure-Rust async ZMQ (no libzmq dependency)
- **tokio 1** (`rt` + `time`) — Single-threaded runtime for the ZMQ server thread
- **bytemuck 1** — Safe vertex buffer casting (`Pod`/`Zeroable`)
- **uuid 1** — UUID generation for stimulus identifiers
- **nvml-wrapper 0.10** — NVIDIA GPU utilisation / VRAM metrics (optional at runtime)
- **sysinfo 0.30** — CPU and RAM metrics for the overlay
- **env_logger 0.11 + log 0.4** — Structured logging with in-process ring buffer for the overlay
- Rust edition 2024

## Platform Support

### Linux (Primary Target)
**Target hardware:**
- **Raspberry Pi 5** — Planned, standard `vc4`/`v3d` KMS drivers
- **NVIDIA Jetson Nano** — Target deployment platform
- **Regular Linux desktop** — Development and testing

**Rendering modes:**
- **DRM/Console mode:** Direct hardware rendering via `VK_KHR_display`, no compositor required (bare-metal)
- **Desktop mode:** Wayland or X11 windowing via winit
- **Auto-detection:** Checks for `DISPLAY` or `WAYLAND_DISPLAY` environment variables

### Windows (Experimental)
- **Desktop mode only:** Win32 windowing via winit
- **Purpose:** Client/server protocol testing and development
- **Always auto-detected:** DRM not available on Windows

## Architecture

### Module Structure (`server/src/`)

| Module | Role |
|---|---|
| `main.rs` | Entry point: auto-detect mode, parse args, spawn ZMQ thread, run backend |
| `ipc.rs` | ZMQ REP server thread; protobuf decode → `handle_request` → encode |
| `proto.rs` | `include!` of prost-generated types from `OUT_DIR` |
| `geom.rs` | Shared 2-D geometry helpers |
| `timing.rs` | `FrameStats`, `FramePhases`, `FrameTick` |
| `log_buffer.rs` | In-process ring buffer (`Arc<Mutex<VecDeque<LogEntry>>>`) fed by `env_logger`; read by overlay |
| `scene/` | `SceneState`, stimulus types, deferred mode |
| `scene/command.rs` | `SceneState::handle_request` — dispatches `Request` to create/mutate/delete |
| `scene/stimulus/primitive_shapes.rs` | `RectStimulus`, `DiscStimulus`, `EllipseStimulus` |
| `scene/stimulus/grating/` | Grating stimulus: params, Vulkan pipeline, tessellation, proto mapping |
| `render/render_state.rs` | `RenderState` — shared Vulkan resources and per-frame logic for both backends |
| `render/tess.rs` | lyon tessellation: Rect/Circle/Ellipse → triangle meshes |
| `render/overlay.rs` | egui overlay UI: System, Frame Timing, Stimuli, IPC Log, Benchmarks, Server Log panels |
| `render/system_metrics.rs` | `MetricsSampler` — CPU/RAM (sysinfo) + GPU utilisation/VRAM (nvml-wrapper, optional) |
| `render/system_info.rs` | `SystemInfo`, `ClockSource` — display geometry, hostname, backend, vblank source |
| `render/benchmark.rs` | `BenchmarkState` — frame-time stress testing |
| `render/vk/` | Shared Vulkan code: `VkContext`, `VkPipeline`, `VkGratingPipeline`, `GpuBuffers`, `render_frame` |
| `render/vk/egui/` | Custom Vulkan egui renderer (not egui-wgpu) |
| `render/drm/` | DRM/console backend: `VK_KHR_display` surface, libinput keyboard, vblank wait |
| `render/winit_vk/` | Desktop backend: winit window, `VK_KHR_surface` via ash-window |
| `lib.rs` | Exposes all modules as a library crate for integration tests |

### Wire Protocol

`proto/vstimd/v1/` — four files; all messages are protobuf-encoded over ZMQ REQ/REP.

- `common.proto` — `Vec2`, `Color`, `Backend`, `DrawMode`, `StimulusType`
- `service.proto` — `Request`/`Response` envelope, `ErrorCode`, `SystemTarget`
- `stimuli_2d.proto` — all create/mutate/query messages for 2-D stimuli
- `system.proto` — scene-wide commands and `QueryServerInfoResponse`

**Creation (system target → returns handle + UUID in Response):** `CreateRect`, `CreateCircle`, `CreateEllipse`, `CreateGrating`

**Generic stimulus commands (handle > 0):** `SetEnabled`, `SetName`, `Delete`, `SetPosition`, `SetOrientation`, `SetFillColor`, `SetAlpha`, `SetOutlineColor`, `SetOutlineWidth`, `SetDrawMode`, `SetRectSize`, `SetDiscRadius`, `SetEllipseSize`, `QueryStimulus`

**Grating-specific commands:** `SetGratingPhase`, `SetGratingSf`, `SetGratingContrast`, `SetGratingWaveform`, `SetGratingMask`, `SetGratingDriftSpeed`, `SetGratingDriftDecoupled`, `SetGratingDriftAngle`, `SetGratingForeColor`, `SetGratingBackColor`, `SetGratingOpacity`

**System commands (system target):** `SetBackground`, `SetDeferredMode`, `DeleteAll`, `SetAllEnabled`, `QueryServerInfo`, `ListStimuli`

All `Create*` requests accept an optional `id` (client-supplied UUID string; server generates one if empty) and `name` (human-readable label). `ListStimuliResponse` returns `StimulusEntry` rows with `handle`, `type`, `enabled`, `id`, and `name`.

### Python Client (`client/python/`)

See `client/python/README.md` for the full API reference.

`vstimd.Connection` wraps a `zmq.REQ` socket and exposes two sub-clients. Protobuf stubs live in `vstimd/_proto/`, generated from the four `proto/vstimd/v1/` files.

```python
from vstimd import Connection

with Connection() as conn:
    h = conn.stimuli.create_rect(x=-200, y=0, width=300, height=200, r=1.0, g=0.0, b=0.0)
    conn.stimuli.set_enabled(h, False)
    conn.stimuli.delete(h)
    info = conn.system.query_server_info()
```

```
client/python/
  vstimd/
    __init__.py          # exports Connection, ServerInfo, exceptions, psychopy
    _connection.py       # ZMQ REQ socket + protobuf wrapper
    _proto/              # generated stubs (common, service, stimuli_2d, system)
    stimuli/             # StimuliClient: create_rect/circle/ellipse/grating, mutations, query
    system/              # SystemClient: query_server_info, set_background, deferred mode, …
    psychopy/            # PsychoPy compatibility layer (Window, Rect, Circle, GratingStim)
    exceptions.py        # error code → Python exception mapping
  examples/
    flash_rects.py       # create two rects, flash them alternately
    interactive_grating.py
  tests/
    unit/                # API signature tests (no server required)
    e2e/                 # full round-trip tests (null renderer or real server)
```

### DRM Backend — Jetson Hardware Notes

The **Jetson Nano** target hardware has a split architecture similar to the Jetson Orin:

| DRM node | Hardware | Role |
|---|---|---|
| `card0` / `renderD128` | nvgpu (`13e00000.host1x`) | GPU, Vulkan runs here |
| `card1` / `renderD129` | display controller (`13800000.display`) | Scanout, KMS connectors |

`VkPhysicalDeviceDrmPropertiesEXT` reports `hasPrimary=false` / `hasRender=false` because
the Vulkan driver (nvgpu) does not associate itself with any DRM node.
**`VK_EXT_acquire_drm_display` does not work on this hardware** — the extension
requires the Vulkan device and the display controller to be the same hardware node.

The working approach is **`VK_KHR_display`**: the Vulkan driver enumerates displays
directly without a DRM fd, as long as no other process holds the display. GDM/Xorg
must not be running. If GDM is the display manager, disable it:
```bash
sudo systemctl disable --now gdm
```

### Working status

**Both backends:**
- Rect, Circle, Ellipse stimuli — tessellated with lyon, solid-colour Vulkan pipeline
- Grating stimulus — analytical fragment shader (WGSL → SPIR-V via naga), supports sin/square/sawtooth/triangle waveforms, circle/Gauss/Hann/raised-cosine aperture masks, drift animation, independent drift direction, per-grating fore/back colors and opacity
- All stimuli have server-assigned UUIDs and optional human-readable names
- Deferred mode (atomic multi-stimulus flip on next vsync)
- ZMQ REP server on `tcp://0.0.0.0:5555`
- Null renderer mode (`--null` / `VSTIMD_NULL=1`): ZMQ server only, no display

**DRM/console mode (auto-detected on Linux without display server):**
- Vulkan swapchain via `VK_KHR_display` at native display resolution (FIFO vsync)
- Vblank wait: DRM vblank preferred, falls back to `VK_KHR_present_wait`, then display-timing, then GPU-completion
- Keyboard via libinput: **D** demo stimuli, **ESC** exit, **F1** overlay toggle
- egui overlay: working (same custom Vulkan renderer as desktop mode)

**Desktop/winit mode (auto-detected on Linux/Windows with display server):**
- Vulkan swapchain via `VK_KHR_surface` + `ash-window` (Wayland/X11/Win32)
- Keyboard: **D** demo, **ESC** exit, **F1** overlay toggle, **F3** wireframe toggle, **Alt+Enter** fullscreen
- egui overlay: working — panels for System (CPU/RAM/GPU metrics), Frame Timing (sparkline + phase breakdown), Stimuli (live enable toggles), IPC Log (last 200 commands), Benchmarks, Server Log

### ZMQ Bind Address

The `zeromq` crate resolves the host part as DNS — `tcp://*:5555` (libzmq wildcard convention) will fail. Use `tcp://0.0.0.0:5555` to listen on all interfaces.

### Planned Architecture (see `docs/`)

See `docs/PLAN.md` for the full design and roadmap. Additional planning documents are in the `docs/` directory.

Key architectural decisions already made:
- Stimulus types: flat `enum` with composition (not trait objects or inheritance)
- Position input: keep shared memory (ZMQ latency too high at ~100–400 µs round-trip)
- 2-D and 3-D coexist in one frame (3-D rendered first, 2-D overlaid)
- Render thread must never block or heap-allocate on event emission

### Threading Model

The server has two concurrent threads sharing `Arc<RwLock<SceneState>>`:

| Thread | Lock held | When |
|---|---|---|
| **render** (`render/drm/` or `render/winit_vk/` → `render/vk/frame.rs`) | write → read | Once per frame: write for tessellation bookkeeping, read for draw list |
| **ZMQ server** (`ipc.rs`) | write | One command dispatch at a time (`handle_request`) |

The `RwLock` write lock in `update()` is dropped before `render()` acquires a read
lock, so the ZMQ thread always has a window to process commands between frames.

**`Send + Sync` requirements:**  
`Arc<RwLock<SceneState>>: Send` requires `SceneState: Send + Sync`.  
`SceneState` contains `IndexMap<u32, Animation>` where `Animation` is a plain
enum (same pattern as `Stimulus`). All concrete animation structs are `Send + Sync`
by default since they contain no raw pointers or thread-local state.

**`lib.rs`:**  
A `src/lib.rs` was added to expose `proto`, `scene`, `ipc`, and other modules
as a library crate, enabling `tests/commands.rs` integration tests to call
`SceneState::handle_request` directly without GPU or ZMQ.
